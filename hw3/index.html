Overview

The homework asked us to implement a physic-based path tracer for ray-tracing image rendering. Part 1 is the most basic ray tracing geneation and camera re-projection. Part 2 is then the acceleration with BVH data structure. Part 3 asked us to do direct light illumination. Part 4 extend it to indirect, multiple-bounced global illumination with recursive calls and Russian Roulette termination. The final part asked us to implemnt adaptive sampling for dynamic sample points allocation.

Part 1: Ray Generation and Scene Intersection

The pipeline of ray generation and primitive intersection is as follows:

  • High-level idea: For each pixel (x, y), we generate rays according to the camera parameters. Then, we solve for the intersection between the ray and the primitive surface (either a triangle or a sphere in this assignment). In this way, we can accumulate the radiance along the ray and assign the correct pixel color.
  • Low-level implementation:
    1. For each pixel (x, y), we call raytrace_pixel(x, y) (in src/pathtracer/pathtracer.cpp).
    2. For each pixel, we take multiple samples (ns_aa) using the sampler (in Part 1, we use the pre-implemented GridSampler) to obtain random offsets (dx, dy). After that, we normalize the coordinates using the height and width to ensure the outputs are within [0, 1].
    3. Then, we call the generate_ray() function, which we are asked to implement in Part 1.1 (src/pathtracer/camera.cpp). First, we perform a series of 2D coordinate conversions (e.g., FOV, tangent, and normalization). Then, we compute the vector from the camera center to (x, y, -1) as the ray in camera space. After that, we multiply it by the c2w camera matrix. Once projected into world space, we apply normalization and clipping settings as described in the homework spec. The output of this step is a ray defined in world space.
    4. Next, we call est_radiance_global_illumination(Ray), which triggers bvh->intersect(ray, &isect). Here, we need to check whether the ray hits any primitive (triangles or spheres). Note that checking every primitive is inefficient, and we will address this issue in Part 2.
    5. At this point, we need to implement both the triangle intersection test and the sphere intersection test. For triangles, we use the Möller-Trumbore algorithm (which we will explain in the following part of the report), and for spheres, we solve a quadratic equation.
    6. If there is an intersection, we record the BSDF and the normal at the intersection point. In Part 1, we do not yet fully accumulate radiance, since the current return value is simply normal_shading(isect.n).
  • At this stage, we have completed the core part of the rendering pipeline. We will add acceleration in Part 2, illumination in Parts 3 and 4, and update the sampler in Part 5.

Implementation of the triangle intersection algorithm

All of the implementation is in src/scene/triangle.cpp. Following Lecture 13, we use the Möller-Trumbore algorithm to test whether there is an intersection between a ray O + tD and a triangle plane parameterized as (1 - b1 - b2)P0 + b1P1 + b2P2. The implementation is in my helper function bool Triangle::test(const Ray &r, double &t, double &b1, double &b2).

In the normal case, we can solve for t, b1, and b2 by computing E1, E2, S, S1, and S2, and then following the standard derivation. However, there are several special cases we need to handle carefully:

  1. If the divisor dot(S1, E1) is close to zero (e.g., less than 1e-8), the ray is effectively parallel to the triangle plane, so there is no intersection.
  2. If the resulting b1 or b2 is smaller than zero, or if b1 + b2 > 1, then the ray does not hit the triangle.
  3. If the resulting t is outside the clipped range, we also treat it as a no-intersection case.

Part 2: Bounding Volume Hierarchy

BVH construction implementation

  • Background: In Part 1, the implementation effectively uses a single large bounding box that contains all primitives and always returns true for ray-box intersection. As a result, the intersection cost is O(#rays * #primitives), which becomes very slow when rendering scenes with a large number of primitives.
  • High-level idea: Instead of checking every primitive, we preconstruct a balanced tree-like data structure in advance, which takes roughly O(#primitives * log #primitives). During intersection, we first test the ray against the bounding box at each node, and only perform detailed primitive intersection tests for the relevant leaf nodes. In this way, the overall intersection cost can be reduced to about O(#rays * log #primitives).
  • Low-level implementation:
    1. First, we implement the construct_bvh() function in Part 2.1. In the original code, the primitives are stored in a simple vector-like structure, and the function just returns the head of that list. Now, we instead construct a tree and return the root node. The main idea is that if there are fewer than max_leaf_size primitives (default = 4), we place them into a single leaf node, since that number is small enough to traverse directly. Otherwise, we split the node into two child nodes. The splitting strategy is as follows:
      1. Find the axis with the largest extent among x, y, and z, and use the centroid along that axis as the splitting criterion.
      2. Use std::partition() to divide the primitives around the midpoint. If the partition result is degenerate (that is, the midpoint ends up at the start or end, meaning all primitives lie on the same side), we manually reset the midpoint using the average extent along that axis.
      After finding the midpoint, we recursively build the two subtrees: 
      node->l = construct_bvh(start, mid, max_leaf_size);
node->r = construct_bvh(mid, end, max_leaf_size);
      Overall, this construction process takes O(#primitives * log #primitives) time.
    2. Then, during query time, we first check whether a ray intersects a node’s bounding box. If it does not, we can immediately skip that entire region. The bounding-box test works by computing the entry and exit times of the ray with respect to the slabs on each axis. If the valid time intervals across the three axes do not overlap, we return false; otherwise, we update the range to tighten the intersection bounds.
    3. Finally, we implement BVH traversal for intersection. At the beginning of the function, we use the bounding-box check node->bb.intersect(ray, t0, t1). If the node is a leaf node, we test all primitives inside it and update the BSDF and intersection record accordingly. If the node is an internal node, we recursively traverse both the left and right child nodes.

Part 3: Direct Illumination

Direct lighting implementation

  • High-level idea: After Part 2, we can now do efficient ray tracing, so we know whether rays from each pixel hit a primitive or not. However, up to this point, we have only used synthetic normal shading as the output color. In Part 3, we turn on lighting and implement preliminary illumination, including both uniform hemisphere sampling and importance sampling.
  • Low-level implementation:
    1. In Part 3.1, we implement the diffuse BSDF. Here, f(w_out, w_in) = rho / pi, and the process is straightforward: sample an incoming direction w_in, evaluate f, and use it in the lighting computation.
    2. In Part 3.2, we handle the emission case, that is, when the ray directly hits a light source and there are no bounces involved. In this case, we simply return isect.bsdf->get_emission();
    3. In task 3.3, we consider the case where light bounces off an object before reaching the camera. In this task, we implement uniform hemisphere sampling using Monte Carlo estimation in estimate_direct_lighting_hemisphere(). Specifically, we take N samples and average them. For each sample, we randomly sample an incoming light direction w_i, then check whether the corresponding shadow ray hits an emitting surface using bvh->intersect(shadow_ray, &light_isect). If it does, we compute the cosine term and follow the reflection equation from the homework spec to accumulate the outgoing radiance.
    4. While the previous method works, many sampled directions contribute nothing because they do not hit a light source. Therefore, in the next task (3.4), we implement importance sampling. The main difference is that instead of sampling directions uniformly over the hemisphere, we sample directly from the light sources. For delta lights, we take one sample; for area lights, we take multiple samples. We use light->sample_L to sample a point on the light, then cast a shadow Ray. If the shadow ray is not blocked before reaching the light, we accumulate the corresponding outgoing radiance.

Comparison between uniform hemisphere sampling and light sampling

Comparing the two methods, light sampling performs much better than uniform hemisphere sampling for direct lighting. The main reason is that uniform hemisphere sampling wastes many samples on directions that do not hit any light source, so a large portion of the computation contributes nothing to the final estimate. As a result, the rendered image is usually noisier (low-quality) for the same sampling budget. In contrast, light sampling draws samples directly from the light sources, so most of the samples are actually useful. This makes the estimator much more efficient and produces cleaner soft shadows with fewer samples.


Part 4: Global Illumination

Global illumination implementation

  • Background: In Part 3, we implemented direct lighting, which only accounts for light that travels directly from a light source to a surface and then to the camera (i.e., one bounce). However, in the real world, light bounces multiple times between surfaces before reaching the camera, creating indirect illumination effects such as color bleeding, soft ambient lighting, and realistic shadows. Part 4 extends our path tracer to handle these multiple bounces, implementing full global illumination.
  • High-level idea: The key is to implement recursive path tracing in at_least_one_bounce_radiance(). For each ray-surface intersection, we first compute the direct lighting contribution using one_bounce_radiance(). Then, we sample a new random direction from the BSDF, cast a bounce ray in that direction, and recursively call at_least_one_bounce_radiance() to gather indirect lighting. This recursion continues until we hit max_ray_depth or terminate early via Russian Roulette sampling. The main function est_radiance_global_illumination() orchestrates the process by handling edge cases for different max_ray_depth values and the isAccumBounces flag.
  • Low-level implementation:
    1. In est_radiance_global_illumination(), we handle different cases based on max_ray_depth. For m=0, we return only zero-bounce radiance (emission). For m=1, we return emission plus one-bounce radiance if accumulated, or only one-bounce if not accumulated. For m>=2, we call at_least_one_bounce_radiance() to handle recursive bounces, either adding all bounces (accumulated mode) or filtering to show only the mth bounce (non-accumulated mode).
    2. In at_least_one_bounce_radiance(), we first compute L_out as the one-bounce radiance. If we're in accumulated mode, we always include this direct lighting. In non-accumulated mode, we only include it if the current depth matches max_ray_depth - 1. Then, if the ray depth has not exceeded max_ray_depth, we apply Russian Roulette with continuation probability 0.65 to decide whether to trace further bounces. If we continue, we sample a new direction wi from the BSDF, create a bounce ray, and recursively call at_least_one_bounce_radiance(). The indirect contribution is then accumulated using the rendering equation: L_out += bsdf_sample * indirect * cos_theta / (pdf * continuation_prob).
    3. Russian Roulette is a variance reduction technique that probabilistically terminates rays early while keeping the estimator unbiased. By dividing by the continuation probability, we compensate for the rays that were terminated, ensuring the expected value remains correct. This significantly improves rendering efficiency by avoiding deep recursion for rays that contribute very little to the final image.

Sample-per-pixel rate analysis

To analyze the effect of the number of samples per pixel on rendering quality, I rendered CBbunny.dae with varying sample rates: 1, 2, 4, 8, 16, 64, and 1024 samples per pixel, using 4 light rays for each. As expected, with very few samples (s=1 or s=2), the image is extremely noisy because Monte Carlo estimation requires many samples to converge. As we increase the sample count to 4, 8, and 16, the noise gradually decreases and the image becomes smoother. At 64 samples, the image looks reasonably clean, and at 1024 samples, the noise is almost completely eliminated. This demonstrates the fundamental trade-off in Monte Carlo rendering: more samples lead to better quality but also increase rendering time proportionally. The goal is to find a balance between quality and performance, which is exactly what adaptive sampling (Part 5) aims to optimize.


Part 5: Adaptive Sampling

Adaptive sampling implementation

  • Background: In the previous parts, we used a fixed number of samples per pixel (e.g., 1024 samples). However, not all pixels require the same number of samples to converge. Some regions of the image (like smooth surfaces or constant-color areas) converge quickly with few samples, while other regions (like complex lighting boundaries or high-frequency details) need many more samples to reduce noise. Adaptive sampling optimizes rendering by detecting when a pixel has converged and stopping early, thus saving computation time without sacrificing image quality.
  • High-level idea: The core idea is to track the variance of the samples taken so far for each pixel. Periodically (every samplesPerBatch samples), we compute a confidence interval. If the interval is tight enough (i.e., I <= maxTolerance * mean), we consider the pixel converged and stop sampling early. Otherwise, we continue sampling up to num_samples. This is implemented in raytrace_pixel() by maintaining running statistics s1 and s2 to compute the sample mean and variance on the fly.
  • Low-level implementation:
    1. In raytrace_pixel(), we initialize two accumulators: s1 for the sum of illuminance values and s2 for the sum of squared illuminance values. For each sample, we generate a camera ray, call est_radiance_global_illumination() to get the radiance, and accumulate both the radiance and its illuminance (radiance.illum()) into s1 and s2.
    2. Every samplesPerBatch samples (checked using modulo), we compute the sample mean mean = s1 / n and sample variance variance = (1.0 / (n - 1)) * (s2 - (s1 * s1) / n). From the variance, we compute the standard deviation sigma = sqrt(variance), and then the 95% confidence interval half-width I = 1.96 * sigma / sqrt(n).
    3. If I <= maxTolerance * mean, the pixel has converged, and we break out of the sampling loop early, recording the actual number of samples used in sampleCountBuffer. Otherwise, we continue sampling. This way, pixels that converge quickly use fewer samples, while challenging pixels get the full budget. The result is both faster rendering and a more efficient use of computational resources.
    4. One tricky detail is handling the variance calculation carefully to avoid numerical issues. We use the formula variance = (1.0 / (n - 1)) * (s2 - (s1 * s1) / n) to compute variance in a numerically stable way. Also, we only start checking for convergence after at least samplesPerBatch samples have been taken, ensuring we have enough data for a meaningful estimate.
